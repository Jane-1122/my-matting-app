# Jing's Video Matting Studio - 云端部署说明

本文介绍如何将项目部署到 Render.com，使网站可通过公网访问。

---

## 一、前置条件

1. **GitHub 账号**：用于托管代码
2. **Render 账号**：免费注册 https://render.com
3. **项目已推送到 GitHub**：将本仓库 push 到你的 GitHub 仓库

---

## 二、部署步骤（Render.com）

### 步骤 1：推送代码到 GitHub

```bash
cd my-matting-app
git init
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/你的用户名/你的仓库名.git
git push -u origin main
```

### 步骤 2：在 Render 创建 Blueprint

1. 登录 https://dashboard.render.com
2. 点击 **New** → **Blueprint**
3. 连接你的 GitHub 仓库
4. Render 会自动识别根目录的 `render.yaml`
5. 选择仓库后点击 **Apply**

### 步骤 3：配置环境变量（首次部署会提示）

部署时 Render 会提示填写以下变量，**先跳过**，等后端部署完成后再补全：

| 服务 | 变量名 | 说明 |
|------|--------|------|
| matting-backend | CORS_ORIGINS | 前端地址，例如 `https://matting-frontend.onrender.com` |
| matting-frontend | NEXT_PUBLIC_API_URL | 后端 API 地址，例如 `https://matting-backend.onrender.com/api` |

**操作顺序：**

1. 先让两个服务完成首次部署（可暂时留空或填占位符）
2. 部署完成后，在 Dashboard 中复制 **matting-backend** 的 URL（如 `https://matting-backend-xxxx.onrender.com`）
3. 进入 **matting-frontend** → **Environment**，添加：
   - `NEXT_PUBLIC_API_URL` = `https://matting-backend-xxxx.onrender.com/api`（替换为你的后端实际 URL）
4. 进入 **matting-backend** → **Environment**，添加：
   - `CORS_ORIGINS` = `https://matting-frontend-xxxx.onrender.com`（替换为你的前端实际 URL）
5. 分别对两个服务点击 **Manual Deploy** → **Deploy latest commit** 重新部署

### 步骤 4：等待构建完成

- 首次构建约 10–20 分钟（需下载 PyTorch、Node 依赖等）
- 构建完成后，访问前端 URL 即可使用

---

## 三、注意事项

### 1. 内存与性能

- Render 免费版（Starter）内存有限，视频抠像可能较慢
- 若需更好体验，可升级到 Standard 或更高档位

### 2. 冷启动

- 免费服务长时间无访问会休眠，首次请求可能需等待 30–60 秒唤醒

### 3. 通用物品抠像模型

- `general_object` 和 `general_object_hq` 依赖 ONNX 模型：`u2net.onnx`、`isnet-general-use.onnx`
- 需将模型文件放入 `backend/models/` 并提交到 Git，或通过构建脚本在 Docker 中下载
- 若未包含模型，仅人物抠像（RVM）可用

### 4. 上传限制

- Render 单次请求有大小限制，超大视频可能上传失败
- 建议视频控制在 100MB 以内

---

## 四、其他云平台（可选）

除 Render 外，也可考虑：

| 平台 | 说明 |
|------|------|
| **Railway** | 支持 Docker，需分别为 frontend、backend 创建服务 |
| **Fly.io** | 支持 Docker，需分别部署两个应用 |
| **阿里云 / 腾讯云** | 使用容器服务（如 ACK、TKE）部署 docker-compose |

部署思路类似：前端配置 `NEXT_PUBLIC_API_URL` 指向后端，后端配置 `CORS_ORIGINS` 允许前端域名。

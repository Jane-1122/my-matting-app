# Jing's Video Matting Studio - 云端部署说明

本文介绍如何将项目部署到 Render.com，使网站可通过公网访问。

---

## 一、前置条件

1. **GitHub 账号**：用于托管代码
2. **Render 账号**：免费注册 https://render.com
3. **项目已推送到 GitHub**：将本仓库 push 到你的 GitHub 仓库

---

## 二、部署步骤（Render.com）

### 步骤 1：推送代码到 GitHub

```bash
cd my-matting-app
git init
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/你的用户名/你的仓库名.git
git push -u origin main
```

### 步骤 2：在 Render 创建 Blueprint

1. 登录 https://dashboard.render.com
2. 点击 **New** → **Blueprint**
3. 连接你的 GitHub 仓库
4. Render 会自动识别根目录的 `render.yaml`
5. 选择仓库后点击 **Apply**

### 步骤 3：配置环境变量（首次部署会提示）

部署时 Render 会提示填写以下变量，**先跳过**，等后端部署完成后再补全：

| 服务 | 变量名 | 说明 |
|------|--------|------|
| matting-backend | CORS_ORIGINS | 前端地址，例如 `https://matting-frontend.onrender.com` |
| matting-frontend | NEXT_PUBLIC_API_URL | 后端 API 地址，例如 `https://matting-backend.onrender.com/api` |

**操作顺序：**

1. 先让两个服务完成首次部署（可暂时留空或填占位符）
2. 部署完成后，在 Dashboard 中复制 **matting-backend** 的 URL（如 `https://matting-backend-xxxx.onrender.com`）
3. 进入 **matting-frontend** → **Environment**，添加：
   - `NEXT_PUBLIC_API_URL` = `https://matting-backend-xxxx.onrender.com/api`（替换为你的后端实际 URL）
4. 进入 **matting-backend** → **Environment**，添加：
   - `CORS_ORIGINS` = `https://matting-frontend-xxxx.onrender.com`（替换为你的前端实际 URL）
5. 分别对两个服务点击 **Manual Deploy** → **Deploy latest commit** 重新部署

### 步骤 4：等待构建完成

- 首次构建约 10–20 分钟（需下载 PyTorch、Node 依赖等）
- 构建完成后，访问前端 URL 即可使用

---

## 三、注意事项

### 1. 内存与性能

- Render 免费版（Starter）内存有限，视频抠像可能较慢
- 若需更好体验，可升级到 Standard 或更高档位

### 2. 冷启动

- 免费服务长时间无访问会休眠，首次请求可能需等待 30–60 秒唤醒

### 3. 通用物品抠像模型

- `general_object` 和 `general_object_hq` 依赖 ONNX 模型：`u2net.onnx`、`isnet-general-use.onnx`
- 需将模型文件放入 `backend/models/` 并提交到 Git，或通过构建脚本在 Docker 中下载
- 若未包含模型，仅人物抠像（RVM）可用

### 4. 上传限制

- Render 单次请求有大小限制，超大视频可能上传失败
- 建议视频控制在 100MB 以内

## 四、Railway 部署（推荐）

### 步骤 1：创建两个服务

1. 在 Railway 新建 Project
2. 添加 **Backend** 服务：连接仓库，Root Directory 设为 `backend`，使用 Dockerfile 构建
3. 添加 **Frontend** 服务：同一仓库，Root Directory 设为 `frontend`，使用 Dockerfile 构建

### 步骤 2：配置环境变量

**Backend 服务：**

| 变量名 | 值 | 说明 |
|--------|-----|------|
| `CORS_ORIGINS` | `*` 或 `https://你的前端.railway.app` | 设为 `*` 时允许任意来源访问，便于分享链接给他人直接使用 |
| `MATTING_VIDEO_MAX_SIDE` | `720`（默认） | 视频处理最大边长，越小越快（如 `480` 可加速） |
| `MATTING_FRAME_SKIP` | `1`（默认） | 仅通用抠像生效：`2`=隔帧处理约 2x 速度（人物抠像 RVM 需逐帧保持时序） |
| `PORT` | （Railway 自动注入，无需设置） | 后端已使用 `${PORT:-8000}` |

**Frontend 服务：**

| 变量名 | 值 | 说明 |
|--------|-----|------|
| `NEXT_PUBLIC_API_URL` | `https://你的后端.railway.app/api` | 后端 API 地址，**必须在构建前设置**（Next.js 构建时内联） |
| `PORT` | （Railway 自动注入，无需设置） | 前端 nginx 已支持动态 PORT |

⚠️ **重要**：`NEXT_PUBLIC_API_URL` 是构建时变量。若部署后修改，需 **重新部署** Frontend 才会生效。

### 步骤 3：生成域名

在各自服务的 Settings → Networking 中生成 Public Domain，然后按实际 URL 更新上述环境变量并重新部署。

## 五、免费部署方案对比

| 平台 | 免费额度 | 限制 | 适合本应用 |
|------|----------|------|------------|
| **Render** | 750 小时/月，免费 Web 服务 | 15 分钟无访问休眠，冷启动 30–60 秒 | ✅ 推荐，真正免费 |
| **Railway** | 试用 $5，之后约 $1/月 | 超出需付费 | ⚠️ 试用后可能超支 |
| **Fly.io** | 仅 7 天或 2 小时试用 | 试用后需绑卡 | ❌ 不适合长期免费 |
| **Hugging Face Spaces** | 免费 CPU/GPU Space | 需改造成 Gradio/Streamlit | ⚠️ 需较大改造 |
| **Vercel + 自建后端** | 前端免费 | 后端需另找 | ❌ 前后端分离复杂 |

**推荐免费方案：Render.com**

1. 注册 https://render.com（免费）
2. 连接 GitHub 仓库
3. 使用项目根目录的 `render.yaml`，一键创建前后端
4. 每月 750 小时，两个服务各约 375 小时，足够个人使用
5. 冷启动时首次访问等 30–60 秒即可

## 六、Railway 常见问题

### 收费说明

- **试用期**：新账号有约 $5 试用额度（约 30 天），用完后进入免费档
- **免费档**：每月约 $1 免费额度，超出需升级 Hobby（$5/月）
- **当前显示**：「30 days or $4.86 left」表示试用期内，剩余约 $4.86 额度

### 使用方式

- **无需本地打开**：前后端部署在 Railway 后 24 小时在线，直接访问前端链接即可使用
- **分享给他人**：把前端链接（如 `https://matting-frontend-production.up.railway.app`）发给别人，对方打开即可使用，无需安装

### 红色警告说明

- 历史中的 **FAILED** 部署不影响当前运行，当前 **COMPLETED** 即表示服务正常
- 若新部署失败，可点击「View logs」查看构建日志排查

## 七、其他云平台（可选）

除 Render、Railway 外，也可考虑：

| 平台 | 说明 |
|------|------|
| **Fly.io** | 支持 Docker，需分别部署两个应用 |
| **阿里云 / 腾讯云** | 使用容器服务（如 ACK、TKE）部署 docker-compose |

部署思路类似：前端配置 `NEXT_PUBLIC_API_URL` 指向后端，后端配置 `CORS_ORIGINS` 允许前端域名。
